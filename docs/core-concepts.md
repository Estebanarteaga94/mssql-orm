# Core Concepts

`mssql-orm` is a code-first ORM for Rust applications that target Microsoft SQL Server. The model starts in Rust structs, not in a database introspection step, and the runtime path keeps metadata, query structure, SQL Server compilation and Tiberius execution in separate crates.

The main flow is:

```text
Entity -> Metadata -> Query AST -> SQL Server SQL -> Tiberius -> Row -> Entity
```

For typed projections and raw SQL, the last step can materialize a DTO instead of the full entity:

```text
Entity or DTO metadata contract -> SQL Server result row -> FromRow -> Entity or DTO
```

This document describes the implemented mental model. For the repository inventory behind these claims, see [repository-audit.md](repository-audit.md).

## Workspace Boundaries

The workspace is split by responsibility:

- `mssql-orm-core` defines contracts, metadata, SQL Server type mapping, neutral row/value types and shared errors.
- `mssql-orm-macros` generates metadata and trait implementations from derives.
- `mssql-orm-query` carries query AST types and builder primitives. It does not generate SQL strings.
- `mssql-orm-sqlserver` compiles query and migration structures into SQL Server syntax.
- `mssql-orm-tiberius` owns connection, execution, row adaptation, transactions and operational concerns over Tiberius.
- `mssql-orm-migrate` owns snapshots, diff operations and migration filesystem helpers.
- `mssql-orm-cli` exposes migration commands.
- `mssql-orm` is the public facade crate and the recommended consumer entry point.

Consumers normally import:

```rust
use mssql_orm::prelude::*;
```

The root crate reexports the public API while preserving the internal boundaries. Query building should stay as AST construction, SQL emission should stay in `mssql-orm-sqlserver`, and database I/O should stay in `mssql-orm-tiberius`.

## Entities

An entity is a Rust struct that represents a persisted table row. `#[derive(Entity)]` generates:

- an implementation of the `Entity` contract,
- static `EntityMetadata`,
- generated column symbols such as `User::email`,
- `FromRow` materialization for the entity fields.

Example:

```rust
use mssql_orm::prelude::*;

#[derive(Entity, Debug, Clone)]
#[orm(table = "users", schema = "dbo")]
struct User {
    #[orm(primary_key)]
    #[orm(identity)]
    id: i64,

    #[orm(length = 180)]
    email: String,

    active: bool,
}
```

The generated metadata is the bridge between Rust and the rest of the ORM. It records table, schema, columns, primary keys, indexes, foreign keys, rowversion columns, computed columns, rename hints and policy-generated columns.

Column symbols are not field reads. `User::email` is a typed reference to the database column and is used to build predicates, ordering and projections.

## Metadata

`EntityMetadata` and related types live in `mssql-orm-core`. They are intentionally neutral:

- they do not depend on Tiberius,
- they do not contain SQL strings generated by the query builder,
- they can be consumed by query building, migrations and DDL generation.

Important metadata contracts include:

- `Entity`
- `EntityMetadata`
- `ColumnMetadata`
- `EntityColumn`
- `IndexMetadata`
- `ForeignKeyMetadata`
- `SqlTypeMapping`
- `SqlValue`
- `Row`
- `FromRow`
- `Insertable`
- `Changeset`

The metadata is also the input for code-first migrations. `ModelSnapshot::from_entities(...)` turns entity metadata into migration snapshots, and the migration crate computes diffs against prior snapshots.

## DbContext And DbSet

`DbContext` is the application-facing unit that owns a shared connection and typed sets. `#[derive(DbContext)]` is used on a struct whose fields are `DbSet<T>` values.

```rust
use mssql_orm::prelude::*;

#[derive(DbContext, Debug, Clone)]
struct AppDb {
    pub users: DbSet<User>,
}
```

The derived context provides connection helpers such as `connect(...)`, `connect_with_config(...)`, `from_connection(...)`, `from_shared_connection(...)`, `health_check()`, `transaction(...)` and `save_changes()`.

`DbSet<T>` is the typed boundary for an entity. It exposes CRUD, query building and experimental tracking:

- `find(key)`
- `insert(model)`
- `update(key, changeset)`
- `delete(key)`
- `query()`
- `query_with(select_query)`
- `find_tracked(key)`
- `add_tracked(entity)`
- `remove_tracked(&mut tracked)`

The public CRUD and tracking routes are currently centered on simple primary keys. Composite primary key metadata exists, but complete public persistence workflows for composite keys are not implemented.

## Query AST

The public query builder produces `mssql-orm-query` structures such as:

- `SelectQuery`
- `CountQuery`
- `InsertQuery`
- `UpdateQuery`
- `DeleteQuery`
- `Expr`
- `Predicate`
- `OrderBy`
- `Join`
- `Pagination`

For reads, `DbSetQuery<T>` wraps a `SelectQuery` and exposes ergonomic methods:

```rust
let users = db
    .users
    .query()
    .filter(User::active.eq(true))
    .order_by(User::email.asc())
    .take(20)
    .all()
    .await?;
```

At this stage, the query crate still only describes intent. It does not know how to quote SQL Server identifiers, choose `@P1` parameter names or emit `OFFSET/FETCH`.

That separation matters because `mssql-orm-query` is a structural layer, while `mssql-orm-sqlserver` is the SQL Server dialect layer.

## SQL Server Compilation

`mssql-orm-sqlserver` compiles query AST values into `CompiledQuery` values. A compiled query contains:

- the SQL Server SQL string,
- ordered `SqlValue` parameters.

The compiler handles SQL Server-specific syntax:

- bracket-quoted identifiers such as `[dbo].[users]`,
- parameters named `@P1`, `@P2`, ...,
- `OUTPUT INSERTED.*` for insert/update routes,
- `OFFSET ... FETCH` pagination rules,
- joins,
- projection aliases,
- migration DDL.

For example, a filtered query built from `User::active.eq(true)` becomes a parameterized SQL Server statement. The value is carried as a parameter, not interpolated into the SQL string.

`mssql-orm-sqlserver` does not execute SQL. It only compiles.

## Tiberius Execution

`mssql-orm-tiberius` adapts `CompiledQuery` to the Tiberius driver. Its responsibilities include:

- parsing SQL Server connection strings,
- opening connections,
- binding `SqlValue` parameters,
- executing commands and queries,
- converting Tiberius rows into the neutral `Row` contract,
- supporting transactions, health checks, timeouts, tracing, slow-query options, retry options and optional pooling.

The public facade uses this layer through `SharedConnection`, `DbContext`, `DbSet`, `RawQuery` and `RawCommand`.

Tiberius is treated as the low-level driver, not as the ORM. It should not own entity metadata, query AST construction or migration diff logic.

## Row Materialization

Rows come back through the neutral `Row` trait from `mssql-orm-core`. Types are materialized through `FromRow`.

For entities, `#[derive(Entity)]` generates `FromRow` for the actual Rust fields in the entity. For DTO projections or raw SQL, users can implement `FromRow` manually:

```rust
use mssql_orm::prelude::*;

struct UserListItem {
    id: i64,
    email: String,
}

impl FromRow for UserListItem {
    fn from_row<R: Row>(row: &R) -> Result<Self, OrmError> {
        Ok(Self {
            id: row.get_required_typed::<i64>("id")?,
            email: row.get_required_typed::<String>("email")?,
        })
    }
}
```

This is the final step of the core read path:

```text
SQL Server row -> MssqlRow -> Row -> FromRow -> Entity or DTO
```

## Writes

Writes use explicit payload contracts:

- `Insertable<E>` extracts values for insert.
- `Changeset<E>` extracts changed values for update.

Derives exist for both:

```rust
#[derive(Insertable)]
#[orm(entity = User)]
struct NewUser {
    email: String,
    active: bool,
}

#[derive(Changeset)]
#[orm(entity = User)]
struct UpdateUser {
    email: Option<String>,
    active: Option<bool>,
}
```

`DbSet::insert(...)` and `DbSet::update(...)` convert those payloads into query AST values, then the SQL Server compiler emits SQL, then the Tiberius adapter executes it and materializes returned rows.

## Projections

`all()` and `first()` materialize full entities. Typed SQL projections are a separate route:

```rust
let rows = db
    .users
    .query()
    .select((User::id, User::email))
    .all_as::<UserListItem>()
    .await?;
```

Projection aliases are part of the AST through `SelectProjection`. Columns receive stable aliases by default. Expressions require explicit aliases so `FromRow` can read the result by name.

This differs from mapping in memory after `all()`: a projection changes the emitted `SELECT`, while `map` transforms entities after full rows have already been loaded.

## Raw SQL

Raw SQL is the explicit escape hatch for SQL Server queries and commands that do not fit the public query builder yet.

```rust
let rows = db
    .raw::<UserListItem>("SELECT id, email FROM dbo.users WHERE active = @P1")
    .param(true)
    .all()
    .await?;

db.raw_exec("UPDATE dbo.users SET active = @P1 WHERE id = @P2")
    .params((false, 7_i64))
    .execute()
    .await?;
```

Raw SQL still uses ordered parameters and `FromRow`, but it does not apply ORM-level `tenant` or `soft_delete` filters. The caller must write those predicates explicitly.

## Migrations

Migrations use the same entity metadata pipeline:

```text
EntityMetadata -> ModelSnapshot -> diff -> MigrationOperation -> SQL Server DDL
```

The CLI can scaffold migrations, consume current snapshots, call a snapshot exporter binary, generate `up.sql`, generate reversible `down.sql` when possible and assemble database update scripts.

The current migration artifact MVP is SQL/snapshot based:

- `up.sql`
- `down.sql`
- `model_snapshot.json`

`migration.rs` is explicitly deferred and should not be treated as an implemented migration API.

## Entity Policies

Entity policies are code-first extensions that contribute cross-cutting model behavior.

Implemented policy areas include:

- `audit = Audit` as metadata/schema columns through `#[derive(AuditFields)]`,
- `soft_delete = SoftDelete` with runtime delete semantics and read visibility,
- `tenant = CurrentTenant` with opt-in tenant filtering and fail-closed behavior.

Important limits:

- audit columns are metadata/schema only,
- audit columns do not become Rust fields on the entity,
- audit columns do not generate column symbols such as `Todo::created_at`,
- audit insert/update autoloading through an `AuditProvider` is not implemented,
- soft-delete automatic filters apply to the root entity of a `DbSetQuery`, not every joined entity,
- raw SQL does not apply policy filters automatically.

## Current Limits

These are intentional limits of the current implementation:

- SQL Server is the only supported database target.
- Public persistence workflows are centered on simple primary keys.
- `Tracked<T>` and `save_changes()` are experimental.
- `db.transaction(...)` is not supported on contexts created from a pool until a physical connection can be pinned for the whole closure.
- Navigation properties and inferred joins are not implemented.
- Automatic table aliases for repeated table joins or self-joins are not implemented.
- High-level typed aggregate builders are not implemented.
- Runtime `AuditProvider` autoloading is not implemented.
- `migration.rs` is not implemented.

When documenting or using the project, treat these as unavailable unless the code and audit are updated first.
